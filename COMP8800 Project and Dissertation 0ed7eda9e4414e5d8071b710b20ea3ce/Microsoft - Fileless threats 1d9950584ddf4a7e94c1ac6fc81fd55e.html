Fileless Threats: An In-Depth Perspective

Fileless threats may seem mysterious at first glance, as they suggest that a threat does not use a file, such as a backdoor residing only in a machine's memory. However, the term "fileless" is used in a fairly broad context. In some cases, these threats do actually require files to function.

How do Fileless Threats Work?

First and foremost, it is essential to understand that attacks often have multiple stages: execution, persistence, information theft, etc. Some parts of the attack chain may be fileless, while others may interact with the file system in various ways.

Categorization of Fileless Threats: An In-Depth Study

Categorizing fileless threats is an imperative necessity, given the diversity and increasing sophistication of these attacks. By understanding these categories, appropriate countermeasures can be better defined. Let's take a closer look at the detailed types of fileless threats:

1. Type I: No File Activity

In this category, threats do not use any files during their operation. They run entirely in memory, making detection particularly difficult for traditional antivirus solutions.

  • Illustrative Example: Let's imagine a scenario where malware exploiting the EternalBlue vulnerability, known for its rapid propagation capability, manages to infiltrate a system. Once inside, it could install a backdoor like DoublePulsar, which nestles in kernel memory without leaving any physical trace.
  • Other Residences: But that's not all. Threats of this type also have the ability to hide elsewhere than in memory. Device firmware is another favorite location. For example, a threat could discreetly install itself in a computer's BIOS or even in a USB device's firmware, thus escaping usual controls.

2. Type II: Indirect File Activity

These threats are more subtle. Even if they do not directly interact with files, they can manipulate or use files in a roundabout way.

  • Illustrative Example: Take the case of Poshspy. Instead of writing directly to a file, this malicious backdoor cleverly uses the WMI repository. The WMI repository itself is stored on a physical file. However, Poshspy manages to periodically execute a malicious command via this repository without directly interacting with the file.

3. Type III: Files Required to Function

This category is interesting because it shows how clever malware can be. Although they rely on files to function, their persistence method remains "fileless."

  • Illustrative Example: Consider Kovter, malware that cleverly manipulates a system's registry. It creates a file association, which when invoked, triggers the execution of a script via mshta.exe. The perverse beauty of this approach is that the malicious script runs without the file itself being opened.

Entry Points for Fileless Threats

The variety of attack vectors makes protection even more complex. Here are some of the ways they can infiltrate:

  • Exploits: Whether they are executable files, documents, Java or Flash, these attack vectors are commonly used. However, exploits can also be based on network data, without the need to interact with physical files.
  • Hardware: Malware has evolved to target the hardware itself. For example, they can infect the firmware of a network card, a hard drive, or even a modern CPU. The latter, with their integrated firmware, can be vulnerable to specific attacks.
  • Execution and Injection: This is one of the most common infection methods. Malware uses scripts, executables, or macros as execution vectors, often exploiting vulnerabilities in commonly used software.

Reflection:

In conclusion, understanding these categories and entry points is essential to developing a robust defense against fileless threats. These attacks represent the future of the threat landscape, and organizations must be prepared to face them with adequate security strategies.
